# Security Model

Tabula Lens is designed with security as a core principle. Learn about the security architecture and best practices.

## Core Security Principles

### Credential Isolation

Database credentials never leave the backend server. This is the fundamental security principle of Tabula Lens.

**Why this matters:**
- Browser-based applications cannot safely store database credentials
- Connection strings in frontend code are exposed to all users
- Database credentials in browser storage are vulnerable to XSS attacks

**How Tabula Lens solves this:**
- Credentials stay on your backend server
- Frontend only receives query results, never credentials
- Backend enforces database access control

### Authentication

Authentication is handled at the HTTP API level using standard token-based authentication.

**Implementation:**

```javascript
// Backend: Verify authentication
app.get('/api/tabula-lens', async (req, res) => {
  const token = req.headers.authorization?.replace('Bearer ', '')
  const user = await verifyToken(token)
  
  if (!user) {
    return res.status(401).json({ error: 'Unauthorized' })
  }
  
  const data = await tabulaLens.query(req.query)
  res.json(data)
})
```

```jsx
// Frontend: Send authentication
<DatabaseViewer 
  path="/api/tabula-lens"
  getAuthHeaders={() => ({
    'Authorization': `Bearer ${userToken}`
  })}
/>
```

### Authorization

Authorization is enforced on the backend before database queries are executed.

**Best practices:**
- Implement row-level security in your database
- Use database users with limited permissions
- Enforce access control based on user roles
- Validate table access permissions

## Security Best Practices

### Backend Security

1. **Environment Variables**
   ```bash
   # Never commit credentials to version control
   DATABASE_URL="postgresql://user:password@localhost:5432/mydb"
   JWT_SECRET="your-secret-key"
   ```

2. **Database Users**
   ```sql
   -- Use read-only users when possible
   CREATE USER tabula_lens_readonly WITH PASSWORD 'secure-password';
   GRANT SELECT ON ALL TABLES IN SCHEMA public TO tabula_lens_readonly;
   ```

3. **Connection Security**
   ```javascript
   // Use SSL in production
   const tabulaLens = new TabulaLens(process.env.DATABASE_URL, {
     ssl: { rejectUnauthorized: true }
   })
   ```

### Frontend Security

1. **Token Storage**
   ```javascript
   // Store tokens securely (httpOnly cookies recommended)
   // Avoid localStorage for sensitive tokens
   ```

2. **HTTPS Only**
   ```javascript
   // Always use HTTPS in production
   const endpoint = 'https://api.example.com/api/tabula-lens'
   ```

3. **Input Validation**
   ```javascript
   // Validate all inputs on the backend
   const { table, page, pageSize } = req.query
   
   if (!isValidTable(table)) {
     return res.status(400).json({ error: 'Invalid table' })
   ```

## Common Security Pitfalls

### ❌ Don't: Expose Database Credentials

```javascript
// NEVER do this
const tabulaLens = new TabulaLens('postgresql://user:password@host/db')
// This exposes credentials in frontend code
```

### ✅ Do: Use Backend Proxy

```javascript
// Backend: Secure credential storage
const tabulaLens = new TabulaLens(process.env.DATABASE_URL)

// Frontend: Query through API
<DatabaseViewer path="/api/tabula-lens" />
```

### ❌ Don't: Bypass Authentication

```javascript
// NEVER do this
app.get('/api/tabula-lens', async (req, res) => {
  // No authentication check!
  const data = await tabulaLens.query(req.query)
  res.json(data)
})
```

### ✅ Do: Always Authenticate

```javascript
// Always verify authentication
app.get('/api/tabula-lens', async (req, res) => {
  const user = await authenticate(req)
  if (!user) return res.status(401).json({ error: 'Unauthorized' })
  
  const data = await tabulaLens.query(req.query)
  res.json(data)
})
```

## Error Handling

Security-conscious error handling prevents information leakage:

```javascript
// Generic error messages
try {
  const data = await tabulaLens.query(req.query)
  res.json(data)
} catch (error) {
  // Don't expose internal details
  console.error('Database error:', error) // Log details server-side
  res.status(500).json({ 
    error: 'An error occurred while processing your request' 
  })
}
```

## Security Architecture

### Threat Model

Tabula Lens addresses the following security threats:

**1. Credential Exposure**
- **Threat**: Database credentials exposed in frontend code
- **Mitigation**: Credentials never leave backend server
- **Implementation**: HTTP API proxy pattern

**2. Unauthorized Access**
- **Threat**: Unauthorized users accessing database data
- **Mitigation**: Authentication and authorization at API level
- **Implementation**: Token-based authentication with role-based access control

**3. SQL Injection**
- **Threat**: Malicious SQL code injected via query parameters
- **Mitigation**: Parameterized queries and input validation
- **Implementation**: PostgreSQL parameterized queries

**4. Data Leakage**
- **Threat**: Sensitive data exposed in error messages
- **Mitigation**: Generic error messages and server-side logging
- **Implementation**: Structured error handling

**5. Man-in-the-Middle Attacks**
- **Threat**: Interception of data in transit
- **Mitigation**: TLS/SSL encryption
- **Implementation**: HTTPS-only in production

### Security Layers

Tabula Lens implements defense in depth across multiple layers:

**Layer 1: Network Security**
- TLS/SSL encryption for all connections
- HTTPS-only in production
- Secure database connections with SSL

**Layer 2: Authentication**
- Token-based authentication (JWT, OAuth, session-based)
- Secure token storage (httpOnly cookies recommended)
- Token expiration and refresh mechanisms

**Layer 3: Authorization**
- Role-based access control (RBAC)
- Table-level access permissions
- Row-level security (RLS) in database

**Layer 4: Input Validation**
- Parameterized queries to prevent SQL injection
- Input sanitization and validation
- Type checking for all parameters

**Layer 5: Data Protection**
- Sensitive data masking in logs
- Generic error messages
- Secure credential storage

**Layer 6: Monitoring**
- Security event logging
- Audit trails
- Anomaly detection

## Advanced Security Features

### Row-Level Security (RLS)

Implement row-level security in PostgreSQL for fine-grained access control:

```sql
-- Enable RLS on a table
ALTER TABLE users ENABLE ROW LEVEL SECURITY;

-- Create policy for user-specific data
CREATE POLICY user_isolation ON users
  FOR ALL
  USING (user_id = current_user_id());

-- Create policy for admin access
CREATE POLICY admin_access ON users
  FOR ALL
  USING (
    EXISTS (
      SELECT 1 FROM user_roles
      WHERE user_id = current_user_id()
      AND role = 'admin'
    )
  );
```

### Audit Logging

Implement comprehensive audit logging for security events:

```javascript
const auditLogger = {
  logAccess: (userId, table, action) => {
    logger.info('Database access', {
      userId,
      table,
      action,
      timestamp: new Date().toISOString(),
      ip: req.ip,
      userAgent: req.headers['user-agent']
    });
  },
  
  logAuthentication: (userId, success, reason) => {
    logger.info('Authentication attempt', {
      userId,
      success,
      reason,
      timestamp: new Date().toISOString(),
      ip: req.ip
    });
  },
  
  logAuthorization: (userId, table, authorized, reason) => {
    logger.info('Authorization check', {
      userId,
      table,
      authorized,
      reason,
      timestamp: new Date().toISOString()
    });
  }
};

// Use in your API
app.get('/api/tabula-lens', async (req, res) => {
  const user = await authenticate(req);
  
  auditLogger.logAuthentication(user.id, true, null);
  
  if (!checkTableAccess(user, req.query.table)) {
    auditLogger.logAuthorization(user.id, req.query.table, false, 'Insufficient permissions');
    return res.status(403).json({ error: 'Access denied' });
  }
  
  auditLogger.logAuthorization(user.id, req.query.table, true, null);
  auditLogger.logAccess(user.id, req.query.table, 'query');
  
  const result = await tabulaLens.query(req.query);
  res.json(result);
});
```

### Rate Limiting

Implement rate limiting to prevent brute force attacks:

```javascript
const rateLimit = require('express-rate-limit');

const apiLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutes
  max: 100, // Limit each IP to 100 requests per windowMs
  message: 'Too many requests from this IP',
  standardHeaders: true,
  legacyHeaders: false,
  handler: (req, res) => {
    logger.warn('Rate limit exceeded', {
      ip: req.ip,
      path: req.path,
      timestamp: new Date().toISOString()
    });
    res.status(429).json({ error: 'Too many requests' });
  }
});

const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 5, // Stricter limit for authentication endpoints
  skipSuccessfulRequests: true
});

app.use('/api/tabula-lens', apiLimiter);
app.post('/api/auth/login', authLimiter);
```

### IP Whitelisting

Implement IP whitelisting for additional security:

```javascript
const ipfilter = require('express-ipfilter');

const allowedIPs = [
  '192.168.1.1',
  '192.168.1.2',
  '10.0.0.0/8' // Allow entire subnet
];

app.use('/api/tabula-lens', ipfilter(allowedIPs, {
  mode: 'allow',
  log: false,
  logLevel: 'deny',
  detectIp: (req) => {
    return req.headers['x-forwarded-for'] || req.connection.remoteAddress;
  }
}));
```

## Incident Response

### Security Incident Response Plan

**1. Detection**
- Monitor security logs for suspicious activity
- Set up alerts for authentication failures
- Monitor for unusual query patterns

**2. Containment**
- Revoke compromised credentials
- Block suspicious IP addresses
- Disable affected accounts

**3. Investigation**
- Review audit logs
- Determine scope of breach
- Identify root cause

**4. Recovery**
- Restore from backups if needed
- Update security measures
- Communicate with stakeholders

**5. Post-Incident**
- Conduct post-mortem
- Update security policies
- Implement lessons learned
